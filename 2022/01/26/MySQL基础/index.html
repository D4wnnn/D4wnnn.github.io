6.事务

在Mysql中,只用InnoDB引擎支持事务

6.1事务是什么?

就是一组逻辑单元,使数据从一种状态到另一种状态

6.2ACID是什么?

  • 原子性

    就是说事务不可分割,要么全成功,要么全失败

  • 一致性

    指的是数据从一种合法状态到另一种合法状态

  • 隔离性

    一个事务的执行不受其他事务干扰

  • 持久性

    数据的改变是永久性的

6.3事务的状态

  • 活动的

    就是正在执行

  • 部分提交的

    事务的操作完成,但是只是在内存中,并没有刷新到磁盘

  • 失败的

    事务在活动的或者部分提交的状态遇到了错误或者认为停止

  • 中止的

    回滚完成时,就是处于中止状态

  • 提交的

    同步到磁盘上

6.4如何使用事务?

显式事务

  1. start transaction或者begin

    注意start transaction后面能接修饰符

    • read only:只读事务
    • read write:读写事务
    • with consistent snapshot:启动一致性读

  2. 一系列事务的操作

  3. 提交事务或者回滚事务

    • 提交:commit

    • 回滚:rollback或者rollback to [savepoint]

      在一个事务中可以存在多个保存点:

      创建保存点savepoint 保存点名称

      删除保存点release savepoint 保存点名称

隐式事务

查看自动提交功能: show variables like 'autocommit';

如何关闭自动提交呢?

set autocommit = OFF;针对DML语句有效,针对DDL语句无效

6.5隐式提交数据的情况

  1. 有一个事务还没提交或者回滚的时候就用begin或者start transaction
  2. 将原来的autocommit设置为ON
  3. 使用LOCK TABLES,UNLOCK TABLES等关于锁定的语句也会隐式提交前面语句所属事务
  4. DDL语句

6.5 数据并发问题

  • 脏写:

    B正在修改数据,A突然来修改B修改过的数据,然后A就commit,按理来说根据数据的持久性,数据不该再变化,但是B这个时候rollback一下,就出现了脏写的问题

  • 脏读:

    还是说B正在修改数据,A突然来读了一下,然后B直接rollback,这个时候A读取的数据就是临时且无效的.

  • 不可重复读:

    大多在并发问题的时候出现

    就是说A读取了一个数据,然后B更新数据,A再次读,数据就不一样了

  • 幻读

    A读取一个数据然后B插入(必须是插入,不是删除)一些新的行,之后A再次读取同一个表就会多出几行.新插入的那些记录成为幻影记录

6.6 SQL中的四种隔离级别

  • READ UNCOMMITTED 读未提交 一般不选此标准
  • READ COMMITTED 读已提交 Oracle默认为此
  • REPEATABLE READ 可重复读 Mysql默认为此
  • SERIALIZABLE 可串行化

上面四种标准都解决的脏写的问题

7.锁

当多个线程并发访问一些敏感数据的时候,我们需要保证这个数据在任何时刻最多只有一个线程在访问,所以引入了锁的概念.

7.1 并发事务访问相同记录

  • 读-读情况:

    无影响

  • 写-写情况:

    会出现脏写的问题,任一隔离界别都不允许这种问题的发生,所以要执行多个未提交事务时需要排队执行.

    当一个事务想对这条记录改动时,首先看看内存有没有与这条记录相关的锁结构,没有就创建.

    在锁结构里面有很多信息,有两个比较重要的属性:

    • trx:当先锁是哪个事务生成的
    • is_waiting:当前事务是否在等待

  • 读-写情况:

    可能出现脏读,不可重复读,幻读

  • 并发问题的解决方案

    • 方案一:读操作利用多版本并发控制MVCC,写操作进行加锁
    • 方案二:读写操作都加锁

7.2 数据操作类型的角度理解S锁与X锁

对于InnoDB引擎,读锁和写锁既可以加在表上,也可以加在行上

  • S锁:共享锁/读锁

  • X锁:排他锁/写锁
